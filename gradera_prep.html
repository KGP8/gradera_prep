05 — Project Stories
STAR format stories
These are your strongest talking points. Know all 3 cold.
Story 1 — Stripe Race Condition (ProCoach)
Most Important
Situation
On ProCoach, we had a production issue where 3% of users were seeing payment failure screens despite their Stripe charge succeeding. Real money was being taken but users thought their enrollment failed.
Root Cause
The frontend was reading enrollment status immediately after the Stripe redirect, but the webhook from Stripe to our backend was arriving 2-4 seconds later — so the enrollment wasn't confirmed in the DB yet when the UI checked.
Action
I implemented a polling fallback — after payment redirect, the frontend polls the enrollment status endpoint up to 3 times with 8-second intervals before showing any failure state. This gave the webhook time to arrive and be processed.
Result
Payment failures dropped from 3% to 0%. Fixed and deployed within 4 hours of identifying the issue. Added the same pattern as a standard across all payment flows.
Story 2 — Real-time WebSocket System (RDR Tech)
Architecture Story
Situation
RDR Tech was an audit and taxation workflow platform. The original system was email-based coordination — teams would email each other when task statuses changed. Completely manual, error-prone.
Challenge
We needed real-time task status updates across all users without page refreshes — and the system needed to handle 1,500+ concurrent active workflows.
Action
Integrated native WebSocket on the frontend — consumed real-time events from the backend. Built a role-based state machine controlling which roles can move tasks between stages. Not just UI access control — actual business logic.
Result
Handles 1,500+ concurrent workflows with zero polling. Replaced the entire email coordination process. Teams get live updates instantly.
Story 3 — Multi-role RBAC Architecture (ProCoach)
Lead-level Story
Situation
ProCoach had 4 user roles — Coach, Learner, Parent, and Admin — all using the same codebase. Each role sees completely different dashboards, routes, and features.
Challenge
How do you architect a single React codebase that serves 4 different product experiences without becoming unmaintainable spaghetti?
Action
Built a JWT-based RBAC system with role-scoped route guards, API-level permissions, and a Context API state layer that exposes only what each role needs. Components are role-aware but not role-polluted — the role check happens at the routing and data layer, not in every component.
Result
500+ active users across all 4 roles on one unified system. New role-specific features can be added without touching other roles' code.
06 — Technical
React & Architecture Questions
Q1How do you manage state in large React applications? +
"My default is Context API + useReducer for most production apps — it keeps state co-located and avoids over-engineering. I've used this across all 5 production apps at Code Stream. For truly global state that many unrelated components need, I'd reach for Zustand or Redux Toolkit — but I start simple and add complexity only when justified. The key is not treating all state the same — server state, UI state, and form state are different problems. I use React Hook Form for form state, TanStack Query patterns for server state, and Context for app-level state."
Mention ProCoach — 4 roles, complex state, Context API held up fine at 500+ users.
Q2How do you approach building a design system from scratch? +
"I start with tokens — spacing scale, color palette, typography — defined as CSS variables or Tailwind config values. Then I build primitive components: Button, Input, Card, Modal. Everything else composes from these. For ProCoach I used ShadCN UI as the foundation — it gives you accessible, unstyled components that you fully own. You take the code into your repo, you're not locked into a library version. Then I layer Tailwind variants on top. The key principle is that components should be composable, not configurable — fewer props, more composition."
This is directly what Gradera wants — ShadCN + Tailwind design system ownership.
Q3When do you use SSR vs SSG vs CSR in Next.js? +
"I choose based on data freshness and SEO needs. SSG for pages where content doesn't change per user — marketing pages, documentation. SSR for pages that need fresh data per request and SEO matters — public profiles, product listings. CSR for authenticated dashboards where SEO doesn't matter and data changes per user. In ProCoach, I deliberately chose SSR for public coach profile pages — they needed to be indexed by Google for coaches to get organic leads. The dashboard behind login is CSR. I didn't apply SSR globally — that would be over-engineering."
Q4How do you structure a React project for scalability? +
"Feature-based folder structure over type-based. Instead of /components, /hooks, /utils — I use /features/enrollment, /features/auth, /features/dashboard. Each feature has its own components, hooks, types, and API calls co-located. Shared code lives in /shared or /lib. This scales because teams can own features independently without merge conflicts. API calls go through a centralized Axios service layer with interceptors for auth tokens and retry logic — one place to change, all features benefit."
07 — Technical
TypeScript Questions
Q5How do you use TypeScript in React — give a real example? +
"TypeScript is non-negotiable for me in production. The biggest value is at API boundaries — I define response types for every API call, so if the backend changes a field name, TypeScript catches it at compile time instead of runtime. For components, I type all props strictly — no any. For the RBAC system in ProCoach, I defined a Role enum and used it as a discriminated union — the TypeScript compiler would error if I forgot to handle a role in a switch statement. For forms with React Hook Form + Zod, I infer the TypeScript types from the Zod schema — single source of truth for validation and types."
Q6What are generics in TypeScript and when do you use them? +
"Generics let you write reusable code that works with multiple types while maintaining type safety. I use them most for API response wrappers — like a generic ApiResponse<T> type where T is the data payload. Also for custom hooks — a useFetch<T> hook that returns typed data regardless of what you're fetching. In the Axios service layer at Code Stream, I typed the get and post methods with generics so every API call returns a properly typed response — no manual casting needed."
08 — Technical
Payments & Webhooks
Q7Walk me through how you handle payment integration end to end. +
"Frontend creates a payment intent via our backend — never directly with Stripe. Backend creates the intent and returns a client secret. Frontend uses Stripe.js to render the payment UI and confirm the payment with the client secret. On success, Stripe redirects to a return URL. Here's the critical part — the frontend CANNOT trust the redirect alone. The actual confirmation happens via webhook: Stripe sends a checkout.session.completed or payment_intent.succeeded event to our backend, which then updates the enrollment status in the DB. The frontend polls for enrollment confirmation before showing success. This is exactly the race condition I solved in ProCoach."
This answer proves you understand the full payment lifecycle — most candidates only know the frontend part.
09 — Technical
Performance Optimization
Q8How did you optimize a slow React dashboard? +
"The RDR Tech dashboard had a performance issue — it managed 1,500+ workflows and the initial load was sluggish. I attacked it in layers. First, route-level code splitting with React.lazy and Suspense — the dashboard bundle was huge because all routes loaded upfront. Splitting reduced initial bundle by ~40%. Second, component memoization — the workflow list was re-rendering on every state update even for components that didn't change. I added React.memo on list items and useMemo for expensive filtered/sorted arrays. Third, virtualization for long lists — instead of rendering all 1,500 items in the DOM, only render what's visible. The combination brought the dashboard to sub-2-second load on real user devices."
10 — Technical
Testing Questions
Q9What do you test and what do you not test? +
"I focus testing on high-risk paths, not coverage numbers. I test: payment flows end to end — a bug here costs real money. Form validation logic — edge cases users hit constantly. RBAC — if a role can access something they shouldn't, that's a security issue. I don't test: simple presentational components with no logic, third-party library internals, or things that change every sprint. With Jest and React Testing Library, I write tests that simulate real user interactions — click a button, expect a state change — not implementation details. Testing implementation details makes tests brittle and maintenance-heavy."

Technical Leadership

Q10How do you mentor junior developers? +
"At Code Stream, I was the most senior frontend person — so I set the patterns others followed. Code reviews were my main mentoring tool. I'd comment not just on what to change but why — explaining the principle behind the feedback. When someone wrote a component that was doing too much, I'd show them the single responsibility principle in context of our actual code, not abstract theory. I also set up the patterns upfront — the Axios service layer, the form validation approach, the folder structure — so juniors had clear examples to follow rather than figuring it out themselves. Good patterns are better mentoring than frequent corrections."
Q11How do you handle technical debt? +
"I distinguish between intentional and unintentional debt. Intentional debt — we know this is a shortcut, we're taking it to ship faster, we'll pay it back — is fine if documented. Unintentional debt — messy code that happened by accident — needs to be caught in code review. In practice I'd spend 20% of sprint capacity on debt reduction — not as a separate cleanup sprint that never happens, but baked into every sprint. The key is making the debt visible — a tech debt backlog item with business impact framing: 'this component is 800 lines and adding features takes 3x longer than it should.'"
Q12Describe how you'd set up a new frontend project from scratch. +
"Next.js with TypeScript as the base — App Router if we need SSR flexibility. Tailwind for styling with a custom config for the design tokens. ShadCN UI for component primitives — copied into the repo, not a dependency. ESLint + Prettier with strict TypeScript config — enforced in CI so no one merges bad code. Absolute imports configured so no ../../../ paths. Feature-based folder structure from day one. Centralized API service layer with typed responses. Husky pre-commit hooks for lint and type check. Vercel for deployment with preview URLs on every PR. This is exactly how I've started every product at Code Stream."
This answer is gold for Gradera — it shows you think at the system level, not just feature level.
